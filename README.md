# Secret Scanning Validity Test — Custom Patterns

This repository was created to test how the `validity` property of a secret scanning alert behaves when the alert is closed, specifically for alerts generated by **custom patterns**.

## Background

The `validity` field on secret scanning alerts can be set to `active`, `inactive`, or `null` (to clear the override) via the [Update a secret scanning alert](https://docs.github.com/en/enterprise-cloud@latest/rest/secret-scanning/secret-scanning#update-a-secret-scanning-alert) API. The question under investigation: **does closing an alert (e.g., as "revoked") automatically change the validity property?**

## Setup

### 1. Repository creation

A public repository was created under the `oodles-noodles` org with secret scanning enabled.

```bash
gh api --method POST /orgs/oodles-noodles/repos \
  -f name="secret-scanning-validity-test" \
  -f visibility="public" \
  -F auto_init=true

gh api --method PATCH /repos/oodles-noodles/secret-scanning-validity-test \
  -f "security_and_analysis[secret_scanning][status]=enabled" \
  -f "security_and_analysis[secret_scanning_push_protection][status]=enabled"
```

### 2. Example secret committed

A file `config/database.yml` was pushed containing two example secrets matching the pattern `MYAPP-SECRET-2026-` followed by 15 alphanumeric characters:

- `MYAPP-SECRET-2026-xK9mP2nQ7rT4wZ`
- `MYAPP-SECRET-2026-bN3jL8vF5hY1cW`

### 3. Custom pattern created

A repo-level custom pattern named **"MYAPP Secret Key"** was created via the API:

```bash
echo '{"patterns":[{
  "name": "MYAPP Secret Key",
  "pattern": "[a-zA-Z0-9]{15}",
  "start_delimiter": "MYAPP-SECRET-2026-",
  "end_delimiter": "\\b"
}]}' | gh api --method POST \
  /repos/oodles-noodles/secret-scanning-validity-test/secret-scanning/custom-patterns \
  --input -
```

The pattern was then published manually via the GitHub UI (the API creates patterns in an `unpublished` state; publishing via API was not straightforward).

### 4. Alerts generated

After publishing, secret scanning detected both secrets and created two alerts (alert #1 and alert #2), each with an initial `validity` of `unknown`.

## Test Procedure (Alert #1)

All API calls target `repos/oodles-noodles/secret-scanning-validity-test/secret-scanning/alerts/1`.

### Step 1 — Confirm initial state

```bash
gh api .../alerts/1 --jq '{state, validity}'
# → { "state": "open", "validity": "unknown" }
```

### Step 2 — Set validity to `active` while alert is open

```bash
echo '{"validity":"active"}' | gh api --method PATCH .../alerts/1 --input -
# → { "state": "open", "validity": "active" }
```

### Step 3 — Close the alert as "revoked"

```bash
echo '{"state":"resolved","resolution":"revoked","resolution_comment":"Simulating secret revocation for validity test"}' \
  | gh api --method PATCH .../alerts/1 --input -
# → { "state": "resolved", "resolution": "revoked", "validity": "active" }
```

**Result: validity remained `active` — closing the alert did NOT change it.**

### Step 4 — Set validity to `inactive` on the closed alert

```bash
echo '{"validity":"inactive"}' | gh api --method PATCH .../alerts/1 --input -
# → { "state": "resolved", "resolution": "revoked", "validity": "inactive" }
```

### Step 5 — Verify final state

```bash
gh api .../alerts/1 --jq '{number, state, resolution, resolved_at, validity, secret_type}'
# → {
#     "number": 1,
#     "resolution": "revoked",
#     "resolved_at": "2026-06-11T12:41:53Z",
#     "secret_type": "myapp_secret_key",
#     "state": "resolved",
#     "validity": "inactive"
#   }
```

## Results Summary

| Step | Action | Validity Before | Validity After |
|------|--------|-----------------|----------------|
| 1 | Alert created by custom pattern scan | — | `unknown` |
| 2 | Set validity to `active` (while open) | `unknown` | `active` |
| 3 | Close alert with resolution `revoked` | `active` | `active` (unchanged) |
| 4 | Set validity to `inactive` (while closed) | `active` | `inactive` |

## Key Findings

1. **Closing an alert as "revoked" does NOT automatically change the `validity` property.** After resolving the alert, validity remained `active`.
2. **The `validity` field can be updated on a closed/resolved alert.** Setting it to `inactive` after the alert was already resolved succeeded without error.
3. **The `validity` field is fully independent of the alert's state and resolution for custom pattern alerts.** It must be explicitly set via the API regardless of whether the alert is open or closed.
4. **Custom pattern alerts start with `validity: unknown`.** Unlike partner pattern alerts where GitHub may auto-verify with the provider, custom patterns have no automated validity checking.

## API Reference

- [Update a secret scanning alert](https://docs.github.com/en/enterprise-cloud@latest/rest/secret-scanning/secret-scanning?apiVersion=2026-03-10#update-a-secret-scanning-alert) — supports `validity` parameter with values `active`, `inactive`, or `null`
